# Security & Legal Audit — localModels

**Date:** 2026-07-02
**Scope:** `C:/Users/Francisco Soltero/Desktop/localModels` (full repo: `README.md`, `HANDOFF.md`, `experiments/`, `inference/`, `research/`, `modelEnv/`, `modelEnvGpu/`)
**Method:** Scout-then-swarm audit — adaptive specialist finders proposed findings across security, supply-chain, licensing, and regulatory dimensions; each finding was independently reviewed by a 3-skeptic panel applying reproduce / exploitability / missing-context lenses, with majority vote determining survival.

---

## 1. Executive summary

This is a solo, local-only Python ML research project (zero git commits at time of audit) with no web server, database, authentication layer, or regulated-data handling anywhere in its own code. Of 15 candidate findings raised by specialist scouts, only 3 survived panel review, and all three are low severity. Twelve findings were refuted, most commonly because trust_remote_code=True in inference/qwen_chat.py was verified to be a functional no-op against the actual cached model snapshot, and because several supply-chain/license findings assumed distribution or CI/production exposure this pre-commit, single-operator sandbox does not have. Overall posture is healthy for the project's current scope; residual forward-looking risks are: (1) no LICENSE/NOTICE file if the repo is ever made public, (2) modelEnvGpu/ (6 GB) is not covered by .gitignore unlike modelEnv/, and (3) no pinned dependency manifest exists, so no CVE scan could be run.

### Low

#### L-1: Training corpus fetched from a mutable branch ref with no integrity check
- File:line: experiments/01_tiny_gpt_baseline/train.py:28 (URL definition), train.py:96 (fetch/write)
- Category: Supply-chain / unpinned remote artifact
- Evidence:
  ```
  DATA_URL = "https://raw.githubusercontent.com/karpathy/char-rnn/master/data/tinyshakespeare/input.txt"
  DATA_PATH.write_text(requests.get(DATA_URL, timeout=30).text, encoding="utf-8")
  ```
- Impact: The URL targets the mutable master branch of a third-party GitHub repo rather than a pinned commit/tag, and the downloaded text is written to disk with no hash/checksum verification. load_data() is reachable, unconditional code invoked from main() — this is not dead code. Impact is capped at low because the content is public-domain plain text consumed only as character data for training (never executed, evaluated, or deserialized as code); a compromised upstream file would silently poison training data with no detection, not enable code execution.
- Remediation: Pin the URL to a specific commit SHA instead of master; optionally verify a known SHA-256 of the expected file content before writing it to disk.

#### L-2: No regulatory framework applicability (SOC 2 / HIPAA / PCI-DSS) — informational
- File:line: C:/Users/Francisco Soltero/Desktop/localModels (repo-wide)
- Category: Regulatory
- Evidence: No HTTP server, routes, database, authentication/authorization layer, multi-tenant boundary, user-identity logging, or persistent storage of conversational/personal data exists anywhere in research/, experiments/, or inference/. inference/qwen_chat.py's conversation history is held only in an in-memory list (conversation = []), never written to disk or transmitted to a third party.
- Impact: No payment data (PCI-DSS N/A), no protected health information (HIPAA N/A), and no customer-facing service processing data on behalf of others (SOC 2 N/A). Documented for audit-trail completeness rather than as a remediation item.
- Remediation: No action required at this time. Revisit if the project's scope changes to expose a network-facing service, persist user data beyond the current process, or process any regulated data class.

#### L-3: No attribution/notice for Qwen model license terms
- File:line: inference/qwen_chat.py:18
- Category: Attribution
- Evidence: pipe = pipeline("text-generation", model="Qwen/Qwen2.5-3B-Instruct", ...) — no LICENSE/NOTICE file exists anywhere in the repo, and no README/comment references the model's license terms.
- Impact: Some Qwen2.5 variants (notably 3B and 72B, unlike the Apache-2.0-licensed 0.5B/1.5B/7B/14B/32B) are distributed under Alibaba's custom Tongyi Qianwen / Qwen license, which historically imposes obligations such as a "Built with Qwen" attribution notice and a commercial-license trigger above 100M MAU. Current exposure is negligible for a solo local script (no redistribution, no product, no MAU), but this is untracked in the repo.
- Remediation: Confirm the current license field on the Qwen/Qwen2.5-3B-Instruct model card and, if it is the custom Qwen license, add a short NOTICE/README entry recording model source, license type, and any required attribution before any redistribution, sharing, or productization.

## 3. Refuted findings appendix

Findings proposed by specialist scouts that did not survive the skeptic panel's majority vote. Included for scope transparency.

### R-1: trust_remote_code=True allows arbitrary Python execution sourced from a remote model repo
- File: inference/qwen_chat.py (medium severity, code-injection/RCE category)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: The locally cached model snapshot's config.json declares "architectures": ["Qwen2ForCausalLM"] with no auto_map key, and the snapshot directory contains zero .py files — transformers only executes remote code when a repo declares auto_map pointing at custom modeling files. Qwen2ForCausalLM is natively implemented in the installed transformers==4.55.0. The flag is a functional no-op for this exact model today. The model ID is also hardcoded, not attacker-controlled. Reduces to a best-practice hygiene nit, not a live medium-severity RCE sink.

### R-2: No authentication/authorization surface exists; sole relevant item is trust_remote_code=True
- File: inference/qwen_chat.py (low severity, authz category)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: Same underlying mechanism as R-1 — no auto_map, no custom .py files in the cached snapshot, so no remote code executes. No attacker-controlled input reaches the model string. Solo-operator local tool with no multi-user/session concept, so classic authz gaps don't apply.

### R-3: trust_remote_code=True enables arbitrary code execution from model repo (insecure-defaults)
- File: inference/qwen_chat.py (medium severity)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: Identical technical refutation to R-1/R-2 (no auto_map, no bundled code, native architecture support, hardcoded non-attacker-controlled model string). Single-user local script with no privilege boundary.

### R-4: Downloaded training corpus has no integrity/hash check
- File: experiments/01_tiny_gpt_baseline/train.py:96 (low severity, missing-integrity-verification)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: Functionally the same underlying fact as confirmed finding L-1; refuted as a duplicate on exploitability/missing-context grounds (one-time download, never executed/deserialized, standard tutorial convention). Superseded by L-1.

### R-5: No pinned dependency manifest exists for either Python environment (lockfile drift)
- File: repo-wide (medium severity, supply-chain category)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: No attacker, attacker-controlled input, or demonstrated CVE was identified — an absence-of-tooling observation, not an exploit path. Repo has zero commits and no CI/publishing pipeline; pip-audit can run directly against installed dist-info metadata without a lockfile.

### R-6: Model loaded with trust_remote_code=True and no pinned revision/hash
- File: inference/qwen_chat.py:16 (medium severity, supply-chain)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: Same config/auto_map analysis as R-1 — no custom code in the resolved snapshot, weights stored as .safetensors (not pickle, so no unsafe-deserialization vector either). Missing revision= pin is a real but much narrower hygiene gap than the "arbitrary RCE" framing claimed.

### R-7: ROCm/PyTorch GPU stack installed from a non-PyPI vendor index, including one sdist
- File: research/notes/hardware.md:24 (low severity, supply-chain)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: repo.radeon.com is AMD's legitimate first-party distribution channel; pip already recorded SHA-256 hashes in direct_url.json at install time. No CI/install-script pathway exists that would blindly re-fetch from this index. The flagged sdist builds into a pure-Python wheel with no native compilation.

### R-8: No CVE/vulnerability scan was performed against either dependency set
- File: modelEnv/ and modelEnvGpu/ (low severity, scanning coverage gap)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: Describes an audit-process limitation (no scanner installed under read-only constraint), not a discovered vulnerability — no CVE named, no attacker or vector identified.

### R-9: No LICENSE or NOTICE file at repository root
- File: repo root (low severity, license category)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: Repo has zero commits and no git remote configured — never published or shared. The finding's own impact was explicitly conditional on a future public release that has not occurred.

### R-10: modelEnvGpu/ (6GB+ ROCm/PyTorch venv) is not gitignored, risking accidental vendoring
- File: .gitignore (medium severity, license category)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: No attacker or untrusted input is involved — only the repo owner could trigger the harm (a broad git add -A), and would see the 6GB untracked directory plainly in git status beforehand. Bundled third-party licenses inspected are standard permissive OSS with notices already present. Repo has an explicit "don't commit without asking" policy documented in HANDOFF.md.

### R-11: ROCm SDK meta-packages ship with zero license metadata and no bundled license text
- File: modelEnvGpu/Lib/site-packages/rocm_sdk_devel-7.2.1.dist-info/METADATA (medium severity, license category)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: Missing license metadata is how AMD packages this wheel upstream — not something the repo owner altered or omitted. No redistribution has ever occurred; the finding's own impact section already conceded this is "currently benign."

### R-12: Inference script pulls Qwen2.5-3B-Instruct under a restrictive custom license with no in-repo documentation
- File: inference/qwen_chat.py (medium severity, license category)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: .gitignore already excludes models/, data/, and weight file extensions, structurally preventing accidental redistribution of model weights via git. Superseded by confirmed finding L-3, affirmed at low severity with tighter scoping.

### R-13: No repo-level LICENSE or NOTICE file to govern the project's own code or declare third-party attributions
- File: repo root (low severity, attribution category)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: Duplicate of R-9's underlying fact. Third-party assets are consumed by runtime reference/download, never vendored or redistributed. Superseded by L-3 for the Qwen-specific angle.

### R-14: trust_remote_code=True enables arbitrary code execution and is unnecessary for this model (high severity)
- File: inference/qwen_chat.py:21 (high severity, unsafe-idiom category)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: Highest-severity restatement of R-1's claim; refuted on identical grounds — verified cached snapshot has no auto_map and no custom .py files, so the flag is presently inert. Attack requires compromise of Alibaba's official Hugging Face org account, a low-likelihood event with no current live payload.

### R-15: GPU stack installed via direct wheel URLs outside PyPI, with no lockfile — untracked by standard vulnerability scanners
- File: modelEnvGpu dist-info direct_url.json files (medium severity, supply-chain / missing manifest)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: No CI/CD, Dependabot, or scanning infrastructure exists in this repo to be "blinded." Repo is pre-first-commit with research/notes/hardware.md already documenting the install rationale, and each wheel's SHA-256 is already captured in its direct_url.json.

### R-16: No dependency manifest of any kind; two mismatched Python/torch stacks maintained only by convention
- File: modelEnv/, modelEnvGpu/ whole-repo (low severity, framework-misconfiguration)
- Panel vote: 1 stands / 2 refute
- Refutation reasons: HANDOFF.md and research/notes/hardware.md already document the exact versions and rationale for both venvs in writing. The GPU env's missing transformers/peft is a deliberately tracked, in-progress TODO, not silent drift. No current code path shares transformers-dependent logic across both venvs.
